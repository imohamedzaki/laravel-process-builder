# Master Implementation Prompt: Laravel Process Builder

You are a senior Laravel package architect, PHP static-analysis engineer, React/TypeScript developer, and developer-tools specialist.

Your task is to design and implement a production-quality open-source Laravel package named:

# Laravel Process Builder

The package must provide a visual interface that allows Laravel developers to design application processes by visually connecting Laravel components such as routes, middleware, form requests, controllers, actions, services, models, events, jobs, notifications, and responses.

The product should feel conceptually similar to the Joget Process Builder, but it must generate clean, native, maintainable Laravel code rather than introducing a proprietary application runtime.

Do not build this as a normal Laravel application.

Build it as a standalone reusable Composer package with an embedded Workbench application for development, testing, and preview.

# 2. Core Product Objective

The package must allow a developer to build a Laravel application flow visually.

A process may look like this:

```text
HTTP Route
    ↓
Middleware
    ↓
Form Request
    ↓
Controller
    ↓
Action
    ↓
Model Operation
    ↓
Event
    ↓
Queued Job
    ↓
API Response
```

Example:

```text
[POST /orders]
        ↓
[auth:sanctum]
        ↓
[StoreOrderRequest]
        ↓
[OrderController@store]
        ↓
[CreateOrderAction]
        ↓
[Order::create]
        ↓
[OrderCreated]
        ↓
[SendOrderConfirmation]
        ↓
[OrderResource — HTTP 201]
```

The package must have two clearly separated operating modes.

## Mode A: Project Explorer

This mode scans and displays existing Laravel routes and controllers.

Existing application files must be treated as read-only.

The package may inspect existing files but must never modify them automatically.

## Mode B: Process Builder

This mode allows users to create new Laravel processes visually.

Only files explicitly managed and generated by Laravel Process Builder may be modified by the package.

The package must clearly identify generated files using:

```php
/**
 * This file is managed by Laravel Process Builder.
 * Manual changes may be overwritten.
 */
```

Never overwrite an existing non-managed application file.

# 17. Generated Code Strategy

Generated code must be clean, readable, and idiomatic Laravel.

For the MVP, generated controllers should be thin.

Business logic should be placed in Action classes.

Example generated route:

```php
<?php

declare(strict_types=1);

/**
 * This file is managed by Laravel Process Builder.
 * Manual changes may be overwritten.
 */

use App\Http\Controllers\ProcessBuilder\OrderController;
use Illuminate\Support\Facades\Route;

Route::middleware(['auth:sanctum'])
    ->post('/orders', [OrderController::class, 'store'])
    ->name('orders.store');
```

Example generated controller:

```php
<?php

declare(strict_types=1);

namespace App\Http\Controllers\ProcessBuilder;

use App\Actions\ProcessBuilder\CreateOrderAction;
use App\Http\Controllers\Controller;
use App\Http\Requests\ProcessBuilder\StoreOrderRequest;
use App\Http\Resources\ProcessBuilder\OrderResource;
use Illuminate\Http\JsonResponse;

final class OrderController extends Controller
{
    public function store(
        StoreOrderRequest $request,
        CreateOrderAction $action
    ): JsonResponse {
        $order = $action->execute($request->validated());

        return response()->json(
            new OrderResource($order),
            201
        );
    }
}
```

Example generated Action:

```php
<?php

declare(strict_types=1);

namespace App\Actions\ProcessBuilder;

use App\Events\ProcessBuilder\OrderCreated;
use App\Models\Order;
use Illuminate\Support\Facades\DB;

final class CreateOrderAction
{
    public function execute(array $data): Order
    {
        return DB::transaction(function () use ($data): Order {
            $order = Order::query()->create($data);

            OrderCreated::dispatch($order);

            return $order;
        });
    }
}
```

Generated code must:

* Use imports.
* Use strict typing.
* Use typed parameters.
* Use typed returns when determinable.
* Follow Laravel conventions.
* Follow PSR-12.
* Avoid unnecessary abstractions.
* Avoid service-locator patterns.
* Avoid static application container calls where constructor injection is clearer.
* Keep controllers thin.
* Generate feature tests where possible.

# 18. Managed Files and Ownership Rules

A file is considered managed only when:

1. It is inside an allowed configured output directory.
2. It contains the exact package-managed marker.
3. It appears in a generation manifest.

Store generation manifests in:

```text
process-builder/manifests/
```

A manifest should contain:

```json
{
    "processId": "01JPROCESS123",
    "processVersion": 1,
    "generatedFiles": [
        {
            "logicalType": "controller",
            "relativePath": "app/Http/Controllers/ProcessBuilder/OrderController.php",
            "sha256": "..."
        }
    ]
}
```

Before overwriting a managed file:

1. Compare its current checksum with the previous manifest.
2. Detect manual modifications.
3. Display a conflict.
4. Refuse overwrite by default.
5. Require an explicit force action to replace manually changed managed code.
6. Always back up the current file before forced replacement.

Never silently destroy changes.

# 19. Preview and Diff Workflow

Generation must use a two-step workflow.

## Step One: Preview

The frontend requests:

```text
POST /process-builder/api/processes/{process}/preview
```

The backend:

1. Validates the process.
2. Compiles files in memory.
3. Does not write anything.
4. Returns generated code.
5. Returns existing-vs-generated diffs.
6. Returns conflicts.
7. Returns a short-lived signed confirmation token.

## Step Two: Generate

The frontend sends:

```text
POST /process-builder/api/processes/{process}/generate
```

With the preview confirmation token.

The backend must verify that:

* The definition has not changed.
* Current files have not changed.
* The preview has not expired.
* The environment is allowed.
* Generation is enabled.

# 35. Architecture Principles

Follow these principles:

1. Existing application code is read-only.
2. Generated code belongs to explicit managed directories.
3. Files are never overwritten silently.
4. Process JSON is the source of truth.
5. Backend validation is authoritative.
6. Frontend validation improves UX only.
7. No raw executable PHP from users.
8. No `eval`.
9. No Regex-based PHP rewriting.
10. Use AST when inspecting PHP syntax.
11. Use DTOs and enums.
12. Use dependency inversion.
13. Keep node implementations modular.
14. Keep controllers thin.
15. Keep the compiler deterministic.
16. Prefer explicit code over magic.
17. Make Git diffs readable.
18. Security must be enabled by default.
19. Generation must require explicit activation.
20. Every destructive operation must be reversible.
